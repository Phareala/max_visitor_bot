Функциональные требования к системе
Проект: «Электронное бюро пропусков»
1. Требования к запуску и первому входу пользователя
Система должна предоставлять пользователю возможность запустить чат-бота на платформе MAX.
При первом запуске бот должен показать пользователю краткое описание сервиса: для чего он нужен, какие действия можно выполнить и какие ограничения есть у системы.
Бот должен отображать дисклеймер о том, что сервис разработан командой хакатона университета и не является официальной функцией платформы.
Система должна объяснять пользователю, что бот предназначен только для оформления разового гостевого пропуска на конкретную дату.
Система не должна предлагать функции, которые находятся за рамками кейса: оформление постоянных пропусков, работу с расписанием, зачётной книжкой, цифровым идентификатором или двухфакторной аутентификацией.
2. Требования к согласию на обработку данных
Система должна запрашивать у пользователя согласие на использование минимальных данных профиля, необходимых для работы сервиса.
Бот должен объяснять, какие данные используются: идентификатор пользователя MAX и отображаемое имя.
Система должна фиксировать факт согласия пользователя в хранилище.
При сохранении согласия система должна записывать:
идентификатор пользователя;
дату и время согласия;
версию текста согласия или документов;
факт принятия условий.
Если пользователь не дал согласие, система не должна позволять ему создавать заявку на пропуск.
3. Требования к ролям пользователей
Система должна поддерживать ролевое разграничение доступа.
В системе должны быть предусмотрены основные роли:
инициатор;
инженер по ИБ / администратор;
технический администратор — только при необходимости.
Система должна назначать новому пользователю роль инициатора по умолчанию.
Система должна проверять роль пользователя при каждом действии, связанном с просмотром, созданием, изменением или обработкой заявки.
Система не должна предоставлять административный доступ через секретную фразу, скрытую команду или общий пароль.
Роль администратора должна быть связана с конкретным пользователем, например через его MAX user ID или заранее настроенную запись в базе данных.
4. Требования к инициатору заявки
Инициатор должен иметь возможность создать заявку на разовый гостевой пропуск.
Инициатор должен иметь возможность просматривать только свои заявки.
Инициатор должен иметь возможность узнать номер и текущий статус своей заявки.
Инициатор должен иметь возможность отменить заявку до того, как по ней принято окончательное решение.
Инициатор должен иметь возможность ответить на запрос уточнения от администратора.
Инициатор должен иметь возможность запросить удаление своих данных.
Инициатор не должен иметь возможность:
видеть заявки других пользователей;
просматривать общую очередь заявок;
подтверждать заявки;
отклонять заявки;
менять статус заявки вручную;
просматривать внутренние комментарии службы безопасности.
5. Требования к созданию заявки
Система должна предоставлять пользователю кнопку или пункт меню для создания разового пропуска.
При создании заявки бот должен вести пользователя по фиксированному сценарию.
Система должна последовательно запрашивать у пользователя следующие данные:
ФИО гостя;
дату визита;
ориентировочное время визита;
корпус или зону посещения;
цель визита.
Система должна использовать кнопки там, где это возможно.
Свободный ввод должен использоваться только для тех данных, которые нельзя удобно выбрать кнопками, например для ФИО гостя и цели визита.
Система должна позволять пользователю исправить ошибочно введённые данные без необходимости проходить весь сценарий заново.
6. Требования к проверке данных
Система должна проверять корректность введённых пользователем данных.
Дата визита не должна быть в прошлом.
ФИО гостя не должно быть пустым.
ФИО гостя не должно состоять только из цифр.
Цель визита не должна быть пустой.
При ошибке ввода бот должен сообщать пользователю, какое именно поле заполнено неверно.
При ошибке ввода бот должен возвращать пользователя к исправлению конкретного поля, а не сбрасывать всю заявку.
Проверки данных должны быть реализованы в логике системы, а не только на уровне сообщений бота.
7. Требования к сводке заявки
После ввода всех данных система должна показать пользователю сводку заявки.
В сводке должны отображаться все данные, которые будут переданы на рассмотрение:
ФИО гостя;
дата визита;
ориентировочное время;
корпус или зона;
цель визита.
После отображения сводки система должна предоставить пользователю два действия:
«Подтвердить»;
«Отменить».
При выборе действия «Отменить» система должна удалить черновик заявки или предложить вернуться к нему позднее. Выбранное поведение должно быть единым для всех пользователей.
При выборе действия «Подтвердить» система должна создать заявку и передать её на рассмотрение.
8. Требования к номеру и статусу заявки
После подтверждения заявки система должна присвоить ей уникальный номер.
Система должна сразу показать пользователю номер созданной заявки.
После создания заявка должна получить статус «На рассмотрении».
Система должна передать созданную заявку в очередь службы безопасности.
Система должна сохранять дату и время создания заявки.
Система должна связывать заявку с инициатором, который её создал.
9. Требования к статусам заявки
Система должна поддерживать жизненный цикл заявки через статусы.
Минимальный набор статусов:
Черновик;
На рассмотрении;
Требуется уточнение;
Подтверждена;
Отклонена;
Закрыта;
Отменена.
Система должна разрешать только допустимые переходы между статусами.
Инициатор должен иметь возможность отменить заявку только до окончательного решения.
Если заявка подтверждена или отклонена, инициатор не должен иметь возможность отменить её.
После подтверждения или отклонения администратор должен иметь возможность закрыть заявку.
Закрытая заявка не должна отображаться в активной очереди.
10. Требования к очереди заявок для администратора
Инженер по ИБ / администратор должен иметь доступ к меню безопасности.
Администратор должен видеть очередь заявок со статусом «На рассмотрении».
Каждая заявка в очереди должна отображаться карточкой.
В карточке заявки должны быть видны:
номер заявки;
дата визита;
ориентировочное время;
корпус или зона;
краткая цель визита;
инициатор заявки по отображаемому имени.
Администратор должен иметь возможность открыть карточку заявки и посмотреть все данные, необходимые для принятия решения.
Администратор не должен иметь доступ к данным пользователей вне контекста рассматриваемых заявок.
11. Требования к обработке заявки администратором
Администратор должен иметь возможность выполнить одно из действий по заявке:
подтвердить заявку;
отклонить заявку;
запросить уточнение.
Все действия администратора должны выполняться через кнопки, а не через произвольные текстовые команды.
После выбора действия система должна изменить статус заявки.
После изменения статуса система должна уведомить инициатора.
После изменения статуса система должна создать запись в журнале событий.
12. Требования к подтверждению заявки
Администратор должен иметь возможность подтвердить заявку, если данных достаточно и визит может быть согласован.
При подтверждении система должна изменить статус заявки на «Подтверждена».
Система должна отправить инициатору уведомление о подтверждении заявки.
В уведомлении должен быть указан номер заявки и новый статус.
Комментарий администратора при подтверждении может быть необязательным.
Факт подтверждения должен быть сохранён в аудит-логе.
13. Требования к отклонению заявки
Администратор должен иметь возможность отклонить заявку.
При отклонении заявки система должна запросить причину отказа.
Причина отказа должна выбираться из короткого списка.
Система должна позволять администратору добавить краткий комментарий при необходимости.
После отклонения система должна изменить статус заявки на «Отклонена».
Система должна уведомить инициатора об отклонении заявки.
Инициатор должен видеть причину отклонения в понятной формулировке.
Система не должна показывать инициатору лишние внутренние сведения службы безопасности.
Факт отклонения должен быть сохранён в аудит-логе.
14. Требования к запросу уточнения
Администратор должен иметь возможность запросить уточнение по заявке.
При запросе уточнения система должна изменить статус заявки на «Требуется уточнение».
Система должна отправить инициатору сообщение с конкретными вопросами.
Инициатор должен иметь возможность ответить на уточнение через кнопки или короткий текстовый ответ.
После получения ответа система должна вернуть заявку в статус «На рассмотрении».
После ответа инициатора заявка должна снова появиться в очереди службы безопасности.
Система должна фиксировать факт запроса уточнения и ответ инициатора в журнале событий.
Система не должна сохранять избыточные данные, если они не нужны для обработки заявки.
15. Требования к отмене заявки
Инициатор должен иметь возможность отменить свою заявку до принятия окончательного решения.
Система должна запрещать отмену заявки, если она уже подтверждена или отклонена.
При отмене заявки система должна изменить статус на «Отменена».
Система должна сохранить факт отмены в журнале событий.
Отменённая заявка не должна отображаться в активной очереди службы безопасности.
16. Требования к закрытию заявки
Администратор должен иметь возможность закрыть заявку после подтверждения или отклонения.
При закрытии заявки система должна изменить её состояние так, чтобы она больше не отображалась в активной очереди.
Факт закрытия заявки должен фиксироваться отдельным событием в аудит-логе.
Закрытие заявки должно завершать её жизненный цикл.
17. Требования к уведомлениям
Система должна уведомлять инициатора о ключевых изменениях по заявке.
Уведомления должны отправляться при следующих событиях:
создание заявки;
присвоение номера заявки;
запрос уточнения;
подтверждение заявки;
отклонение заявки;
закрытие заявки, если это нужно для пользовательского сценария.
В уведомлении должны быть указаны номер заявки и текущий статус.
Текст уведомления должен быть понятным пользователю и не должен содержать внутренних служебных формулировок.
18. Требования к аудит-логу
Система должна вести журнал событий по каждой заявке.
В аудит-лог должны записываться:
согласие пользователя на обработку минимальных данных;
создание заявки;
присвоение номера заявки;
изменение статуса;
запрос уточнения;
ответ инициатора;
подтверждение заявки;
отклонение заявки;
закрытие заявки;
отмена заявки.
Для каждого события система должна хранить:
номер заявки;
тип события;
дату и время события;
пользователя, выполнившего действие;
старый и новый статус, если статус изменился;
комментарий или причину, если они были указаны.
Аудит-лог должен позволять восстановить цепочку событий по номеру заявки.
Аудит-лог не должен содержать лишние чувствительные данные.
19. Требования к техническому администратору
Система может поддерживать роль технического администратора, если команда реализует полноценную поддержку и разграничение доступа.
Технический администратор может иметь доступ к диагностике и настройкам системы.
Технический администратор может просматривать технические ошибки и состояние работы бота.
Технический администратор не должен иметь доступ к чувствительным данным заявок без необходимости.
Логи, доступные техническому администратору, не должны содержать ФИО гостей, подробные цели визита и другие чувствительные поля.
20. Требования к ограничениям по данным
Система должна собирать только минимальный набор сведений, необходимый для оформления разового пропуска.
Система не должна запрашивать:
паспортные данные;
адрес проживания;
финансовые сведения;
иные чувствительные данные, не требуемые для разового пропуска.
Система должна явно показывать пользователю, какие данные передаются на рассмотрение.
Система должна обеспечивать возможность запроса удаления пользовательских данных.
21. Требования к работе через MAX API
Система должна работать через чат-бота на платформе MAX.
Система должна получать события от пользователя через механизмы MAX Bot API.
Система должна отправлять сообщения пользователю через MAX Bot API.
Система должна поддерживать кнопки для основных действий пользователя и администратора.
Для MVP допускается получение событий через polling с использованием метода получения обновлений.
При дальнейшем развитии может быть использован webhook-механизм, если он будет настроен в соответствии с документацией платформы.
