# Электронное бюро пропусков

Чат-бот для мессенджера MAX, позволяющий оформлять разовые гостевые пропуска в корпуса университета прямо из переписки — без звонков и бумажных форм.

---

## Возможности

- **Оформление пропуска** — пошаговый мастер: ФИО гостя, дата, время, зона посещения, цель визита и кастомные дополнительные поля
- **Очередь заявок** — администратор ИБ просматривает поступившие заявки, согласовывает, отклоняет или запрашивает уточнение у инициатора
- **Уведомления в реальном времени** — инициатор получает push при изменении статуса заявки; администратор — при новых заявках и накоплении очереди свыше порога
- **Управление зонами** — технический администратор добавляет, переименовывает и удаляет зоны посещения прямо через бот, без перезапуска
- **Кастомные поля** — для каждой зоны настраиваются дополнительные обязательные или необязательные поля (например, «Гос. номер автомобиля»)
- **Управление ролями** — технический администратор назначает и отзывает роли администраторов ИБ через интерфейс бота
- **Список пользователей** — постраничный просмотр всех пользователей с поиском по имени или ID
- **Журнал аудита** — все действия над заявками фиксируются; возможна выгрузка в CSV
- **Статистика** — счётчики заявок по статусам и зонам за произвольный период или за всё время
- **Автоматическое истечение** — заявки с прошедшей датой/временем автоматически получают статус «Просрочена»

## Роли пользователей

| Роль | Метка | Описание |
|------|-------|----------|
| `initiator` | 👤 Инициатор | Оформляет пропуска и отслеживает свои заявки |
| `admin` | 🛡 Администратор ИБ | Работает с очередью заявок: согласование, отклонение, уточнение |
| `tech_admin` | ⚙️ Тех. Администратор | Полный доступ: управление зонами, полями, ролями, статистикой |

Роли из `.env` (переменные `ADMIN_USER_IDS`, `TECH_ADMIN_USER_IDS`) имеют приоритет над ролями в базе данных.

## Установка и запуск

```bash
git clone <репозиторий>
cd max_visitor_bot

pip install -r requirements.txt

cp .env.example .env
# Отредактируйте .env, заполнив TOKEN и остальные параметры

python main.py
```

При первом запуске автоматически создаётся файл базы данных `visitor_passes.db` и наполняется зонами посещения по умолчанию.

## Конфигурация (.env)

| Переменная | Обязательна | Описание |
|-----------|:-----------:|----------|
| `TOKEN` | ✅ | API-токен бота в мессенджере MAX |
| `BASE_URL` | — | Базовый URL API (по умолчанию `https://platform-api.max.ru`) |
| `ADMIN_USER_IDS` | — | ID администраторов ИБ через запятую |
| `TECH_ADMIN_USER_IDS` | — | ID технических администраторов через запятую |
| `QUEUE_ALERT_THRESHOLD` | — | Порог предупреждения о накоплении очереди (по умолчанию `5`) |

## Архитектура

```
max_visitor_bot/
├── main.py                  — точка входа, monkey-патч reply, глобальный перехватчик /start и /menu
├── database.py              — все операции с SQLite (создание таблиц, CRUD, экспорт)
├── notifications.py         — утилиты push-уведомлений и обработка истёкших заявок
├── scenes/
│   ├── start.py             — запрос согласия на обработку данных, первый вход
│   ├── main_menu.py         — главное меню (маршрутизация по роли пользователя)
│   ├── create_pass.py       — пошаговый мастер оформления пропуска
│   ├── admin_queue.py       — очередь заявок для администратора ИБ
│   ├── user_requests.py     — список заявок инициатора с ответами на уточнения
│   ├── custom_fields_mgmt.py — настройка дополнительных полей заявок по зонам
│   ├── zones_mgmt.py        — управление зонами посещения (CRUD)
│   ├── admin_mgmt.py        — управление ролями администраторов
│   └── user_list.py         — постраничный список пользователей с поиском
└── visitor_passes.db        — база данных SQLite (создаётся автоматически)
```

### Схема базы данных

| Таблица | Назначение |
|---------|-----------|
| `users` | Пользователи: MAX ID, имя, роль, статус согласия |
| `requests` | Заявки на пропуска со всеми полями и статусом |
| `audit_log` | Журнал всех событий: кто, что и когда сделал |
| `custom_fields` | Дополнительные поля, настроенные для каждой зоны |
| `request_custom_fields` | Значения кастомных полей для конкретных заявок |
| `zones` | Зоны посещения (управляются через бот) |

## Жизненный цикл заявки

```
[draft] ──► [review] ──► [approved]
                │
                ├──► [rejected]
                │
                └──► [clarification] ──► [review]

Любой незакрытый статус при истечении даты/времени → [expired]
[approved] / [rejected] → [closed]  (закрывается администратором вручную)
```

---

## Навигация в боте

Команды `/start` и `/menu` работают **из любой сцены** — они немедленно возвращают пользователя в главное меню (или к экрану согласия, если оно ещё не дано). Это обрабатывается глобальным перехватчиком в `main.py` до передачи управления текущей сцене.

## Деплой

Проект рассчитан на запуск через GitHub Actions с автодеплоем на сервер. Сервер синхронизируется командой `git reset --hard origin/main` и перезапускает процесс через `systemd` или аналогичный менеджер процессов.
